# 토큰 절약 데모: "로그인 기능 추가해줘"

실제 시나리오로 에이전트 패턴이 어떻게 토큰을 절약하는지 보여줍니다.

---

## 시나리오 설정

- 프로젝트: Spring Boot 백엔드 + React 프론트엔드
- 요청: "JWT 기반 로그인 기능 추가해줘"
- 코드베이스 크기: 파일 50개, 약 5,000줄

---

## 방법 1: 에이전트 없이 (전통적인 방법)

```
사용자: "JWT 기반 로그인 기능 추가해줘"

메인 Claude (컨텍스트에 점점 쌓임):
│
├── 파일 탐색: src/main/java/** 전체 읽기
│   컨텍스트 +3,000 토큰
│
├── 기존 인증 코드 읽기
│   컨텍스트 +2,000 토큰
│
├── 요구사항 분석 (내부에서)
│   컨텍스트 +1,000 토큰
│
├── 아키텍처 고민 (내부에서)
│   컨텍스트 +1,500 토큰
│
├── UserController.java 수정
│   컨텍스트 +500 토큰
│
├── JwtService.java 생성
│   컨텍스트 +800 토큰
│
├── AuthFilter.java 생성
│   컨텍스트 +600 토큰
│
└── 테스트 (내부에서)
    컨텍스트 +700 토큰

총 컨텍스트: 10,100 토큰 (계속 누적)
이후 대화할 때마다 이 10,100 토큰을 계속 들고 다님
```

**문제점**: 
- 컨텍스트가 빠르게 한계에 도달
- 긴 대화 후반에 가면 초반 내용을 잊어버림
- 모든 세부사항이 메인 Claude 머릿속에 쌓임

---

## 방법 2: 에이전트 패턴 (하네스 방법)

```
사용자: "/build-feature JWT 기반 로그인 기능"

메인 Claude 컨텍스트: [대화 시작, 가벼움]
│
├── Phase 1: planner 에이전트 실행
│   ┌─────────────────────────────┐
│   │ planner 에이전트 (독립 실행) │
│   │ 자체 컨텍스트: 2,000 토큰   │
│   │ 파일 읽기, 요구사항 분석...  │
│   │ 완료 후 소멸                │
│   └─────────────────────────────┘
│   메인에 반환: 요약된 기획서 300토큰
│   메인 컨텍스트: +300 토큰
│
├── Phase 2: 탐색 에이전트 2개 병렬 실행
│   ┌──────────────┐ ┌──────────────┐
│   │ 탐색 에이전트A│ │ 탐색 에이전트B│  ← 동시 실행
│   │ 3,000 토큰   │ │ 2,500 토큰   │
│   │ 완료 후 소멸  │ │ 완료 후 소멸  │
│   └──────────────┘ └──────────────┘
│   메인에 반환: 패턴 요약 400토큰
│   메인 컨텍스트: +400 토큰
│
├── Phase 3: developer 에이전트 실행
│   ┌─────────────────────────────┐
│   │ developer 에이전트 (독립)   │
│   │ 자체 컨텍스트: 4,000 토큰   │
│   │ 코드 작성, 파일 생성...      │
│   │ 완료 후 소멸                │
│   └─────────────────────────────┘
│   메인에 반환: 변경 파일 목록 200토큰
│   메인 컨텍스트: +200 토큰
│
└── Phase 4: tester + reviewer 병렬 실행
    ┌──────────────┐ ┌──────────────┐
    │ tester       │ │ reviewer     │  ← 동시 실행
    │ 2,000 토큰   │ │ 1,500 토큰   │
    │ 완료 후 소멸  │ │ 완료 후 소멸  │
    └──────────────┘ └──────────────┘
    메인에 반환: 이슈 목록 300토큰
    메인 컨텍스트: +300 토큰

메인 컨텍스트 총 증가: 1,200 토큰
(에이전트들이 사용한 총 토큰은 비슷하지만 메인 컨텍스트에 쌓이지 않음)
```

**이점**:
- 메인 컨텍스트가 1,200 토큰만 증가 (vs 10,100 토큰)
- 이후 대화가 계속 가벼움
- 각 에이전트가 자기 역할에만 집중해서 품질도 높음

## 컨텍스트 크기 비교 (10개 기능 개발 후)

```
방법 1 (에이전트 없이):
기능1: 10,100 토큰
기능2: 20,200 토큰  
기능3: 30,300 토큰
...
기능10: 101,000 토큰 ← 컨텍스트 한계 도달!

방법 2 (에이전트 패턴):
기능1: 1,200 토큰
기능2: 2,400 토큰
기능3: 3,600 토큰
...
기능10: 12,000 토큰 ← 아직 여유 있음!
```

## 실제로 느끼는 차이

**에이전트 없이 긴 대화 후반**:
```
사용자: "아까 만든 JwtService에서 토큰 만료 처리가 어떻게 되어 있어?"
Claude: "죄송합니다, 구체적인 구현 내용을 다시 확인해야 할 것 같아요..." 
→ 컨텍스트가 꽉 차서 초반 내용을 잊어버림
```

**에이전트 패턴 사용 후**:
```
사용자: "아까 만든 JwtService에서 토큰 만료 처리가 어떻게 되어 있어?"
Claude: "developer 에이전트 보고서에 따르면 JwtService.java:45에서 
        expiration을 처리하고 있습니다. 확인해볼까요?"
→ 요약된 핵심 정보를 정확히 기억
```

## 언제 에이전트를 쓰고 언제 안 쓰나?

| 상황 | 방법 | 이유 |
|------|------|------|
| 간단한 1줄 수정 | 직접 처리 | 에이전트 실행 오버헤드가 더 큼 |
| 여러 파일에 걸친 기능 개발 | 에이전트 패턴 | 컨텍스트 절약 효과 큼 |
| 코드 탐색 + 구현 + 테스트 | 에이전트 패턴 | 각 전문가가 더 잘 함 |
| 긴 대화가 필요한 복잡한 작업 | 에이전트 패턴 | 컨텍스트 한계를 피할 수 있음 |
| 빠른 질문/답변 | 직접 처리 | 즉각 응답이 더 빠름 |

## 핵심 요약

```
에이전트 패턴 = 위임(delegation)

메인 Claude는 "무엇을 할지" 결정하고 조율
에이전트들은 "어떻게 할지" 실행

팀장이 모든 걸 혼자 하면 지치지만,
팀장이 팀원에게 잘 위임하면 더 많은 일을 할 수 있다.
```
