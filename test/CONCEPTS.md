# 핵심 개념 심층 이해

---

## 1. 에이전트(Agent)란?

에이전트는 **특정 역할을 가진 독립적인 Claude 인스턴스**입니다.

### 에이전트 파일 구조

```markdown
---
name: planner                          ← 에이전트 이름 (고유 식별자)
description: 요구사항을 분석하는 기획자  ← 언제 이 에이전트를 쓸지 설명
tools: Read, Glob, Grep                ← 이 에이전트가 쓸 수 있는 도구들
model: sonnet                          ← 사용할 Claude 모델
color: blue                            ← UI에서 보여줄 색깔
---

여기서부터는 이 에이전트의 "시스템 프롬프트"입니다.
당신은 15년차 시니어 기획자입니다...
```

### 중요한 것: frontmatter (---로 감싼 부분)

| 항목 | 설명 | 예시 |
|------|------|------|
| `name` | 에이전트 식별자 | `planner` |
| `description` | Claude가 자동으로 이 에이전트를 선택할 때 참고 | `요구사항 분석 전문가` |
| `tools` | 허용된 도구 목록 | `Read, Grep, Glob` |
| `model` | 모델 선택 (`sonnet`, `haiku`, `opus`) | `sonnet` |
| `color` | 터미널 색상 | `blue`, `red`, `green` |

---

## 2. 서브에이전트 작동 원리

```
메인 Claude (대화 중)
    │
    │  "planner 에이전트를 실행해서 요구사항 분석해줘"
    │
    ▼
┌─────────────────────────────────────┐
│  planner 에이전트 (새 인스턴스 생성)   │
│                                     │
│  컨텍스트: [agents/planner.md 내용]  │
│           + [주어진 작업 설명]        │
│                                     │
│  메인 대화 히스토리: ❌ 없음          │  ← 핵심! 깨끗한 슬레이트
│                                     │
│  작업 수행 중...                     │
│  파일 읽기, 분석 등                  │
│                                     │
│  완료 → 결과 반환                   │
└─────────────────────────────────────┘
    │
    │  결과를 메인 컨텍스트에 요약해서 돌려줌
    ▼
메인 Claude (결과 받음)
```

### 왜 이게 효율적인가?

**비유로 이해하기**: 
대형 프로젝트에서 팀장(메인 Claude)이 모든 걸 혼자 하면 지칩니다.
대신 팀장은 각 전문가(서브에이전트)에게 "이것 좀 조사해줘" 하고 위임합니다.
전문가는 자기 영역만 파고들고, 결과 보고서만 올립니다.
팀장의 머릿속(컨텍스트)에는 세부사항 대신 핵심 결과만 쌓입니다.

## 3. 도구(Tools) 선택의 중요성

에이전트에게 **필요한 도구만** 줘야 합니다.

```
기획자 에이전트에게 필요한 도구:
✅ Read     - 파일 읽기 (요구사항 문서, 코드 파악)
✅ Glob     - 파일 찾기 (어떤 파일들이 있나 확인)
✅ Grep     - 내용 검색 (특정 패턴 찾기)
❌ Bash     - 명령 실행 (기획자가 서버를 건드릴 필요 없음)
❌ Write    - 파일 생성 (기획자가 코드를 직접 쓸 필요 없음)
❌ Edit     - 파일 수정 (마찬가지)

개발자 에이전트에게 필요한 도구:
✅ Read, Glob, Grep  - 기본 파악
✅ Write    - 새 파일 생성
✅ Edit     - 파일 수정
✅ Bash     - 명령 실행 (빌드, 테스트 등)
```

이렇게 **최소 권한 원칙**을 지키면:
- 실수로 잘못된 파일을 건드리는 것을 방지
- 에이전트의 역할이 명확해짐

## 4. 커맨드(Command) vs 에이전트(Agent)

| 구분 | 역할 | 비유 |
|------|------|------|
| **Agent** | 특정 전문 분야 작업자 | 개별 전문가 (의사, 변호사, 개발자) |
| **Command** | 에이전트들을 조율하는 워크플로우 | 프로젝트 매니저의 진행 계획서 |

**커맨드 파일 구조**:
```markdown
---
description: 기능 개발 전체 워크플로우  ← 커맨드 설명
argument-hint: 기능 설명               ← /build-feature <여기에 뭘 쓰는지>
---

# 기능 개발 커맨드

## Phase 1: 기획
planner 에이전트를 실행해서 요구사항을 분석하세요...

## Phase 2: 개발
developer 에이전트를 실행해서 구현하세요...

## Phase 3: 테스트
tester 에이전트를 병렬로 실행해서 검증하세요...
```

## 5. 병렬 실행 패턴

서브에이전트의 강력한 기능 중 하나는 **병렬 실행**입니다.

```
커맨드가 시작됨
    │
    ├── 에이전트 A (코드 탐색) ────────┐
    ├── 에이전트 B (아키텍처 분석) ────┤ 동시에 실행
    └── 에이전트 C (관련 패턴 탐색) ───┘
                                       │
                                   모두 완료
                                       │
                               결과 취합 및 진행
```

**이점**: 3개 작업을 순차적으로 하면 `3 × 시간`이지만, 병렬이면 `1 × 시간`

## 6. 모델 선택 전략

| 모델 | 속도 | 비용 | 추천 용도 |
|------|------|------|-----------|
| `haiku` | 빠름 | 저렴 | 단순 탐색, 파일 찾기 |
| `sonnet` | 중간 | 중간 | 일반 개발, 코드 리뷰 |
| `opus` | 느림 | 비쌈 | 복잡한 아키텍처 설계 |

**팁**: 탐색용 에이전트는 `haiku`, 설계/리뷰용은 `sonnet`, 중요 결정은 `opus`

## 7. 실제 토큰 절약 계산 예시

### 시나리오: "로그인 기능 추가해줘"

**방법 1 (에이전트 없이)**:
```
메인 Claude가 혼자:
- 코드베이스 전체 파악: +5,000 토큰
- 아키텍처 고민: +3,000 토큰  
- 코드 작성: +4,000 토큰
- 리뷰: +2,000 토큰
= 총 14,000 토큰 (모두 메인 컨텍스트에 쌓임)
```

**방법 2 (에이전트 패턴)**:
```
메인 Claude:
- 에이전트 A (탐색): 별도 컨텍스트 5,000토큰 → 요약 500토큰만 반환
- 에이전트 B (설계): 별도 컨텍스트 3,000토큰 → 요약 300토큰만 반환
- 에이전트 C (구현): 별도 컨텍스트 4,000토큰 → 결과 400토큰만 반환
= 메인 컨텍스트에는 1,200토큰만 추가됨

총 사용 토큰은 비슷하지만, 메인 컨텍스트가 가벼워서
긴 대화를 해도 컨텍스트 한계에 덜 도달함
```
